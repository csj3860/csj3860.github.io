VCS and Git Started

1 . About Version Control

During the period when we develop our project, there are some stages. For example, in the first stage of a simple database application systems, we may design the database managing data and the UI to interact in the second stage.

As a matter of fact, they are what we called “big version”, which means there are large updates. However, the concept of version is not so narrow but means changes to a files or set of files over time, which VCSs( version control systems) record.

The user can roll back to the recorded previous versions. For example, a web page designer who have designed several layouts for a website can use this function to go back to past designs if they like.

2. Distributed Version Control System

There are many VCSs in history used by programmers to assist their tasks. The earliest VCS is Local Version Control System(LCVS). Developer control versions in a crude but simple way, copying files into another directory or a database. Just like :

The cost of this method is high because large storage is needed to place the previous versions.

So they choose another method called Centralized Version Control System (CVCS).This system have a single sever containing all the versioned file and a lot of clients that check out files from it.

There is one fatal drawback in this form: What happen if the sever crashes? This means we will lose everything we have. So merely in security perspective, CVCS is not suitable.

Despite neither CVCS nor LVCS are rejected by history, there is one widely applied system: Distributed Version Control System(DVCS). And git is one of DVCSs and the one of the most famous and popular with programmers from all over the world.

Why DVCS can work well? In DVCS, every clients clone the entire repository and its full history and in this way the whole system can function well even if some severs die.

And in DVCSs, the workflow is very flexible. It allows you to collaborate with others to finish the same project.

3. Summary for the Basic Workflow

First, you should use the command git init to create a repo in a directory.

Second, do your work such as adding several code files and text files in the directory.

Third, use git add to let git track these untracked files.

Fourth, if you modify some files which are tracked, you need to use git add again.

Last, use git commit to commit what you do and create a new version.

4. State of Files in Git

This is a theorical part but not difficult. However, without these basic knowledge things can be unclear.

There are four states:

  • Untracked: The new files adding into the real directory but not in the repo. In other word, they are not tracked by git. One thing we must know is that existing in the repo means the same thing as being tracked.
  • Unmodified: The files which is in the repo and all the changes are staged by git add.
  • Modified: The files in the repo which are edited by user but not staged by git add.
  • Staged: The files that are ready to be committed as a new version.

5. Ignoring Files

Often, some files may not wish to be recorded by git. So we must write the .gitignore file to prevent this.

In “Pro Git”, the writer tells us five points:

  • Blank lines or lines starting with # are ignored.
  • Standard glob patterns work, and will be applied recursively throughout the entire working tree.
  • You can start patterns with a forward slash (/) to avoid recursivity.
  • You can end patterns with a forward slash (/) to specify a directory.
  • You can negate a pattern by starting it with an exclamation point (!).

Remember this 5 points and we may use them later.
